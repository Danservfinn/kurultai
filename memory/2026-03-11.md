   - Two `server.js` processes fighting over port 18790

   - `task-watcher.py` used `from scripts.script_paths import get_script_path`

   - Tasks created when Neo4j unavailable were deleted before completion
   - **Fix:** Added 5-minute grace period to `cleanup_orphan_task_files()`

   - **Status:** Neo4j connected and reconciled (52 tasks tracked)

## Neo4j Fallback Removed (2026-03-11 14:30 EST)

**Change:** Removed filesystem-only fallback from `task_intake.py` when Neo4j is unavailable.

**Before:**
```python
except Exception as e:
    print(f"ERROR: Neo4j unavailable, falling back to filesystem-only: {e}")
    # Filesystem-only fallback (100+ lines of code)
    # Creates task file without Neo4j entry
    return task_id
```

**After:**
```python
except Exception as e:
    print(f"ERROR: Neo4j unavailable — task creation FAILED: {e}")
    print("ACTION REQUIRED: Check Neo4j connectivity before retrying.")
    raise  # Fail immediately, no fallback
```

**Rationale:**
- Filesystem-only tasks caused reconciliation issues (orphan cleanup race conditions)
- Neo4j is the source of truth - tasks MUST exist in Neo4j to be tracked
- Better to fail fast than create degraded state that requires manual cleanup

**Impact:**
- Task creation will FAIL if Neo4j is down (expected behavior)
- No more orphan tasks that exist in filesystem but not Neo4j
- Cleaner state management, easier debugging

**File Modified:** `scripts/task_intake.py` (removed ~120 lines of fallback code)

## Claude Code Authentication Fallback Added (2026-03-11 15:10 EST)

**Issue:** When Claude Code authentication fails (e.g., "Not logged in · Please run /login"), tasks fail immediately without attempting fallback to alternative models.

**Request:** Claude Code should fallback if authentication fails.

**Fix Applied:**

1. **Extended `_is_rate_limit_error()` function** to detect authentication failures:
   - Added patterns: "not logged in", "please run /login", "authentication failed", "auth failed", "invalid credentials", "unauthorized", "401", "token expired", "session expired", "sign in required", "login required", "claude login"

2. **Added explicit auth failure detection** in fallback logic:
   - New `is_auth_failure` variable checks for auth-specific patterns
   - Auth failures trigger fallback with reason "authentication_failed"
   - Fallback priority: auth_failure > rate_limit > stall_timeout

3. **Fallback behavior:**
   - When auth fails, task retries with fallback model (glm-5 via Z.AI or qwen3.5-plus via Alibaba)
   - Tolui excluded (must use abliterated model only - no fallbacks allowed)
   - Fallback events logged to ledger for monitoring

**Configuration:**
- Fallback enabled via `acp.fallback.enabled` in `~/.openclaw/openclaw.json`
- Trigger on rate limit: `acp.fallback.triggerOnRateLimit` (default: true)
- Fallback model: `acp.fallback.fallbackModel` (default: glm-5)

**Files Modified:**
- `scripts/agent-task-handler.py` - Extended auth failure detection and fallback logic

**Testing:**
- Syntax check passed
- Next auth failure will trigger automatic fallback to glm-5

**Note:** This fix addresses the task failure `temujin/normal-1773173000.failed.done.md` which failed with "Not logged in · Please run /login". With this fix, similar failures will automatically retry with the fallback model instead of failing immediately.
