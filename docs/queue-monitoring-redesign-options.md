# Queue Monitoring & Redesign Design Options

Generated by kublai agent critical review and brainstorming session (2026-03-08)

## Executive Summary

The current Kurultai queue system suffers from significant load imbalance issues, with ogedei reaching 11 pending tasks while kublai remains idle at 0. This design analysis presents three validated options for implementing a comprehensive queue monitoring and redistribution system.

## Option A: Enhanced File-Based Monitoring (Recommended)

### Overview
Evolution of current file-based system with continuous monitoring and intelligent redistribution. Maintains compatibility while adding advanced capabilities.

### Architecture
- **Monitoring Service**: Enhanced kublai agent with 60-second monitoring cycles
- **State Tracking**: In-memory cache with periodic filesystem sync
- **Redistribution**: Automatic task movement with capability matching
- **Integration**: Hooks into existing task_intake.py routing logic

### Key Components
- **QueueMonitor Class**: Continuous monitoring with configurable thresholds
- **RedistributionEngine**: Automatic task balancing with skill preservation
- **MetricsCollector**: Performance tracking and alerting
- **ConsistencyManager**: File locking and atomic operations

### Trade-offs
**Pros:**
- Minimal disruption to existing system
- Immediate implementation (1-2 weeks)
- Maintains file-based simplicity
- Preserves all existing routing logic

**Cons:**
- Filesystem I/O overhead at scale (>12 agents)
- Limited scalability compared to alternatives
- Still vulnerable to filesystem corruption
- Performance degrades with high task volumes

### Risk Assessment
| Risk | Severity | Mitigation |
|------|----------|------------|
| Filesystem corruption | HIGH | Regular backups, health checks |
| Race conditions | HIGH | File locking, atomic operations |
| Performance bottlenecks | MEDIUM | Caching, batch operations |
| Alert fatigue | LOW | Suppression, escalation |

### Effort Estimate
- Implementation: Small (2 weeks)
- Complexity: Low-Medium
- Maintenance: Easy

---

## Option B: Hybrid Architecture (Redis + Filesystem)

### Overview
Modern hybrid approach combining Redis for real-time metrics with filesystem for task storage. Best balance of performance and reliability.

### Architecture
- **Redis Layer**: Real-time metrics, alerts, and coordination
- **Filesystem Layer**: Task storage and execution
- **Monitoring Service**: Dedicated service with 30-second cycles
- **Redistribution Service**: Async processing with queue management

### Key Components
- **RedisMetrics**: Real-time queue depth tracking (1-minute resolution)
- **EventDrivenMonitor**: Filesystem event notifications
- **RedistributionQueue**: Async task movement with retry logic
- **AlertingSystem**: Multi-channel notifications with severity levels

### Trade-offs
**Pros:**
- Excellent performance (70% I/O reduction)
- Real-time metrics and alerts
- Better scalability (15-20 agents)
- Built-in reliability patterns

**Cons:**
- Redis dependency adds complexity
- Slight risk of inconsistency between layers
- Requires infrastructure changes
- Longer implementation timeline

### Risk Assessment
| Risk | Severity | Mitigation |
|------|----------|------------|
| Redis unavailability | HIGH | Fallback to filesystem mode |
| Data inconsistency | MEDIUM | Cross-layer validation, transaction logs |
- Performance tuning required | MEDIUM | Load testing, optimization |
- Operational complexity | MEDIUM | Documentation, monitoring |

### Effort Estimate
- Implementation: Medium (4-6 weeks)
- Complexity: Medium
- Maintenance: Moderate

## Option C: Full Event-Driven Architecture

### Overview
Modern event-driven architecture with real-time coordination. Maximum scalability and performance for large-scale deployments.

### Architecture
- **Event Bus**: Central coordination for all queue events
- **Active Agents**: Push-based task distribution
- **State Service**: Dedicated state management service
- **Redistribution Engine**: Event-triggered balancing

### Key Components
- **EventMonitor**: Real-time filesystem event processing
- **TaskDispatcher**: Push-based task distribution
- **StateService**: Centralized state management
- **RedistributionOrchestrator**: Event-driven balancing

### Trade-offs
**Pros:**
- Maximum scalability (20+ agents)
- Real-time responsiveness
- Superior resource utilization
- Natural fault tolerance

**Cons:**
- High complexity and risk
- Major architectural changes
- Long implementation timeline
- Steep learning curve

### Risk Assessment
| Risk | Severity | Mitigation |
|------|----------|------------|
| Implementation complexity | CRITICAL | Phased rollout, extensive testing |
- Debugging challenges | HIGH | Enhanced logging, observability |
- Performance regression | HIGH | Load testing, gradual migration |
- Operational complexity | HIGH | Comprehensive documentation |

### Effort Estimate
- Implementation: Large (3-4 months)
- Complexity: High
- Maintenance: Complex

## Recommendation: Option A (Enhanced File-Based)

**Option A is strongly recommended** for immediate implementation because:

1. **Addresses Critical Issues**: Solves the immediate queue imbalance problem (ogedei at 11 tasks while kublai idle)
2. **Minimal Disruption**: Works with existing file-based queues and task_intake.py
3. **Quick Implementation**: Can be deployed in 1-2 weeks with immediate benefits
4. **Proven Foundation**: Builds on existing, working infrastructure
5. **Clear Migration Path**: Can be enhanced to Option B later as needed

### Implementation Roadmap for Option A

**Phase 1 (Week 1-2): Core Monitoring**
- Implement QueueMonitor class in kublai
- Add 60-second monitoring cycles
- Implement basic redistribution logic
- Add file locking for safety

**Phase 2 (Week 3-4): Enhanced Features**
- Add metrics collection and alerting
- Implement atomic task movement
- Add circuit breakers for failures
- Enhanced logging for debugging

**Phase 3 (Week 5-6): Integration & Testing**
- Integration with existing task_intake.py
- Performance optimization
- Load testing with 7-10 agents
- Documentation and deployment

### Success Metrics

**Immediate Metrics (After Phase 1):**
- Zero queue backups exceeding 5 tasks
- 95% reduction in manual redistribution needs
- Sub-second redistribution trigger time

**Target Metrics (After Full Implementation):**
- Queue balance: <20% variance between agents
- Redistribution latency: <5 seconds
- System throughput: 40% improvement
- Alert accuracy: >95%

### Critical Success Factors

1. **Filesystem Security**: Implement proper file permissions and access controls
2. **Race Condition Prevention**: Robust locking mechanisms are non-negotiable
3. **Performance Monitoring**: Must not degrade task execution performance
4. **Backward Compatibility**: Existing routing logic must continue working
5. **Operational Simplicity**: Should be easier to operate, not more complex

### Risk Mitigation

**Immediate Actions:**
- Implement file permissions this week
- Add basic locking for redistribution
- Create backup procedures before changes
- Implement monitoring during rollout

**Preventative Measures:**
- Continuous performance monitoring
- Regular load testing
- Comprehensive alerting
- Rollback procedures ready

### Conclusion

Option A provides the optimal balance of immediate impact, minimal risk, and clear benefits for the Kurultai system. It solves the critical queue imbalance problem while maintaining system stability and providing a foundation for future enhancements.
