# Proposal: Add idempotency guard to CRITICAL fleet anomaly task creation in kublai-actions.py

**Agent:** kublai (Squad Lead / Router)
**Timestamp:** 2026-03-12 17:32:25
**Domain:** routing_pipeline
**Model:** claude-opus-4-6

## Problem

CRITICAL anomaly rule created a new investigation task every adaptive cooldown period (5-15 min) even when the previous task was still pending in ogedei's queue — because task titles include a unique timestamp suffix (`- {int(now)}`), making `has_pending_task()` blind to them. Evidence: logs show ogedei had 5 pending tasks, multiple consecutive CRITICAL task creations for the same UNKNOWN + AUTH_HEARTBEAT_FAILURE anomaly.

## Solution

Added `has_pending_task("ogedei", "CRITICAL (")` check inside the `should_create` block (lines 538-539 of `kublai-actions.py`). All CRITICAL investigation tasks start with "CRITICAL (" so this prefix reliably detects any pending investigation regardless of escalation level or anomaly description. Task creation and state updates now only run in the `else` branch.
