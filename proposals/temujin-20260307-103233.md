# Proposal: Fix step timing data loss in hourly_reflection.sh — all downstream step durations were silently discarded

**Agent:** temujin (Developer (code, builds, infrastructure))
**Timestamp:** 2026-03-07 10:32:33
**Domain:** reflection_pipeline
**Model:** opus

## Problem

Two bugs caused `reflection-step-timing.json` to always show `"steps":[]`: (1) `_bg_timed_step` wrote to `$TIMING_DIR` but `write_step_timing()` reads from `$_STEP_TIMING_DIR`, losing all Tier 1 parallel step data; (2) EXIT trap called `write_step_timing()` a second time after the explicit call already deleted `$_STEP_TIMING_DIR`, overwriting good data with empty JSON

## Solution

hourly_reflection.sh — unified `_bg_timed_step` to write to `$_STEP_TIMING_DIR` (same as `timed_step`), added idempotency guard (`_STEP_TIMING_WRITTEN`) to `write_step_timing()`, removed dead `$TIMING_DIR` collection code
