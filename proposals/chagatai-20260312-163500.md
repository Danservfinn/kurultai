# Proposal: Fleet Failure Triage Runbook

## Problem

The Kurultai has a 74% fleet-wide failure rate (23/31 tasks in 2h) with 38% of failures classified as "unknown" — meaning the system cannot even diagnose what's going wrong. The ESCALATION_PROTOCOL tells operators WHEN to escalate throughput collapse, but not HOW to diagnose root causes. Operators and agents waste time investigating without a structured diagnostic path.

**Evidence:**
- 300 failures in 7 days: unknown (114), timeout (85), claude_code_crash (80)
- Median failure duration: 55s (suggests process crashes, not timeouts)
- No existing runbook covers fleet-wide failure triage

## Solution

Created `docs/fleet-failure-triage.md` — a data-driven diagnostic runbook with:

1. **Quick Triage** (< 2min): Duration-based and category-based decision trees
2. **Category Diagnostics**: Actionable steps for all 6 failure categories (unknown, timeout, crash, auth, rate_limit, model_error)
3. **Escalation Thresholds**: When to escalate vs. self-heal
4. **Historical Patterns**: Reference table of past incidents and resolutions

Cross-referenced from ESCALATION_PROTOCOL.md's Throughput Collapse row.

## Expected Impact

- **Reduce diagnosis time** from ad-hoc investigation to structured 2-minute triage
- **Improve "unknown" classification** by directing operators to add new patterns to `failed-task-review.py`
- **Faster recovery** from fleet-wide failure events
- **Capability score improvement** for chagatai (demonstrates writer throughput)
