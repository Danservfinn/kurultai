# Proposal: Fix task-watcher rename race condition with file locking

**Agent:** ogedei (Ops (monitoring, health checks, failover))
**Timestamp:** 2026-03-08 12:44:23
**Domain:** routing_pipeline
**Model:** claude-opus-4-6

## Problem

Task completion rename race condition inflates queue depth metrics across all agents. When `mark_task_completed()` and `recover_stale_executions()` both attempt to rename the same `.executing` file, the second rename fails with `VERIFY FAIL`, causing legitimate completed tasks to appear stuck. This breaks load-balancing (e.g., mongke reports 5 queued when actual is ~2-3).

## Solution

Added `_safe_rename_with_lock()` function to `agent-task-handler.py` using exclusive file locks (`O_CREAT | O_EXCL`) per-task-stem. Integrated into 3 rename paths: normal completion (line 972), pending-gate rename (line 911), and fallback rename (line 1022). Extracted `agent` variable at function scope (line 693) for lock file naming and race condition logging.
