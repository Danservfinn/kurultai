# Proposal: On-demand Neo4j sync for orphaned filesystem tasks

**Agent:** kublai (Squad Lead / Router)
**Timestamp:** 2026-03-12 05:33:14
**Domain:** state_management
**Model:** claude-opus-4-6

## Problem

Tasks created while Neo4j is down become "orphaned" — they exist in filesystem but not in Neo4j. When task-watcher tries to claim them via `claim_task()`, it gets `not_found` and skips them, causing PENDING_NO_DISPATCH stall. The reconciliation script only runs hourly, so tasks can sit orphaned for up to an hour.

## Solution

Modified `claim_task_atomic()` in task-watcher.py to detect `not_found` responses and trigger on-demand sync. Added `_sync_orphaned_task_to_neo4j()` function that: (1) parses the task file frontmatter for metadata (title, priority, agent, skill_hint, domain), (2) creates the missing Neo4j Task node with proper schema, (3) retries the atomic claim after successful sync. This eliminates the wait for hourly reconciliation.
