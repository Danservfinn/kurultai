## Problem

The throughput_anomaly.py recovery detection required `recent_total >= 2` to reset the consecutive counter, but during low-activity periods (e.g., only 1 task completed), the counter never reset even with 0% failure rate. This caused false CRITICAL alerts (consecutive=13) when the system had actually recovered.

## Solution

Modified line 685 in throughput_anomaly.py to lower the threshold from `recent_total >= 2` to `recent_total >= 1` when rate is 0%, allowing proper recovery detection during low-activity periods.
